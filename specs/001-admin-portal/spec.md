## User Scenarios & Testing *(mandatory)*

### User Story 1 — 运营人员处理异常订单与重试生成任务（Priority: P1）

运营人员登录后台后，可以查看所有订单列表，按状态筛选出「失败」订单，进入订单详情查看生成任务的逐章日志，判断失败原因后手工触发重试，并在备注中记录处理过程。

**Why this priority**: 论文生成是核心商业承诺（生成成功率 ≥ 99%），失败订单如无法快速处理会直接损害用户体验和退款风险；这是上线第一天即需要的运营能力。

**Independent Test**: 仅部署订单模块后，运营人员能够登录后台、查看订单列表、查看详情与生成任务日志、手工重试一个失败任务并验证状态变为「执行中」。

**Acceptance Scenarios**:

1. **Given** 运营人员已登录，**When** 在订单列表按状态筛选「失败」，**Then** 仅展示生成失败的订单，并可按时间排序。
2. **Given** 运营人员打开某失败订单详情，**When** 查看「生成任务」Tab，**Then** 展示该订单按章生成的进度与每章报错信息。
3. **Given** 某订单处于失败状态，**When** 运营人员点击「重新触发生成」，**Then** 系统将该任务重新加入后台生成队列，订单状态更新为「生成中」。
4. **Given** 运营人员在订单详情写入备注并保存，**When** 该操作完成，**Then** 备注内容持久化，且操作日志记录「谁在何时写入了备注」。

### User Story 2 — 运营人员维护学校格式模板（Priority: P1）

运营人员收到用户通过客服上传的学校格式规范文档后，在后台新建该学校的格式模板，上传 .dotx 文件并配置参考文献规范、标题层级等，启用后用户在步骤 1 可搜索到该学校。

**Why this priority**: 格式模板是用户完成步骤 1 的必要条件，缺少常见高校模板会直接导致用户无法继续流程；这也是客服接收用户反馈后的核心后台动作。

**Independent Test**: 仅部署格式模板模块后，运营人员能够新建一条学校模板、上传 .dotx、启用，并验证用户端步骤 1 可搜索到该学校名。

**Acceptance Scenarios**:

1. **Given** 运营人员进入「格式模板列表」，**When** 搜索某学校名称，**Then** 仅展示名称匹配的模板条目（含启用/停用状态）。
2. **Given** 运营人员点击「新增模板」，**When** 填写学校名称、学历、上传 .dotx 并点击保存，**Then** 新模板以「停用」状态入库，不影响已上线流程。
3. **Given** 运营人员将某模板「启用」，**When** 前端用户在步骤 1 按该校名称搜索，**Then** 该校出现在搜索结果中可供选择。
4. **Given** 某用户反馈「未找到学校」并提交了学校名称与规范文档，**When** 运营在「用户提交」列表中找到该条记录，**Then** 可查看用户上传的文档、将状态标记为「已补充」或「已忽略」，并可关联到已新建的模板。

### User Story 3 — 管理员管理后台账号与角色（Priority: P1）

超级管理员可以创建后台账号（运营、客服、只读等角色），分配角色权限，并查看操作日志，确保每条关键操作（订单状态变更、模板上下架、用户禁用）都可追溯。

**Why this priority**: 安全与审计是上线必须项；不同角色（运营 vs 客服 vs 只读）的权限隔离可防止误操作，操作日志是合规与排查的基础。

**Independent Test**: 仅部署权限模块后，超级管理员可创建一个「运营」账号，登录该账号验证只能访问运营相关菜单，并在操作日志中看到创建记录。

**Acceptance Scenarios**:

1. **Given** 超级管理员进入「账号管理」，**When** 创建一个角色为「客服」的新账号，**Then** 该账号只能访问被授权菜单（如订单查看、用户信息查看），无法访问配置类菜单。
2. **Given** 运营人员执行了「模板启用/停用」操作，**When** 超级管理员查看操作日志，**Then** 日志记录包含：操作者账号、操作时间、操作内容（启用/停用了哪条模板）。
3. **Given** 超级管理员将某账号禁用，**When** 该账号再次尝试登录，**Then** 登录被拒绝并提示账号已停用。

---

### User Story 4 — 运营人员管理用户与风控（Priority: P2）

运营人员可以按用户 ID 或注册时间查找用户，查看其订单历史，对违规用户执行禁用或单日生成上限限流，所有操作自动留痕。

**Why this priority**: 用户风控是防止滥用（如单账号批量生成）的必要手段，但非首日上线阻塞项。

**Independent Test**: 仅部署用户模块后，运营人员可搜索到某用户、查看其订单数，并将该用户禁用，验证被禁用用户前台登录被拒绝。

**Acceptance Scenarios**:

1. **Given** 运营人员在「用户列表」按注册时间筛选，**When** 选择某日期范围，**Then** 仅展示该时段注册的用户（脱敏显示手机/邮箱）。
2. **Given** 运营人员打开某用户详情，**When** 查看，**Then** 展示该用户订单总数、完成数、失败数及最近订单列表。
3. **Given** 运营人员对某用户执行「禁用」，**When** 操作完成，**Then** 该用户的前台登录请求被拒绝，操作日志记录此次禁用。

---

### User Story 5 — 运营人员配置查重与引用核对 API（Priority: P2）

运营人员可以在后台配置第三方查重 API 的地址和密钥，切换查重服务商（如从万方切换到知网），并可配置引用核对接口开关，无需发布代码即可完成切换。

**Why this priority**: 查重是步骤 6 核心能力，但服务商可配置是技术隔离的运营需求，支持在不发版情况下切换。

**Independent Test**: 运营人员可修改查重 API 密钥并保存，验证前台步骤 6 查重请求走更新后的配置。

**Acceptance Scenarios**:

1. **Given** 运营人员在「查重 API 配置」页输入新的服务商地址和密钥，**When** 保存，**Then** 系统后续查重请求使用新配置，旧配置不再生效。
2. **Given** 运营人员关闭「引用核对」开关，**When** 用户在步骤 6 触发引用核对，**Then** 前台提示「该功能暂时关闭」，不向已关闭的接口发起请求。

---

### User Story 6 — 运营人员查看数据统计看板（Priority: P2）

运营人员可以在统计看板查看订单量、各步骤转化率、生成失败原因分布和收入汇总，支持按日/周/月切换，用于产品优化决策。

**Why this priority**: 数据统计是产品迭代的决策依据，但不影响核心生成和运营流程，可在 P0/P1 后补充。

**Independent Test**: 仅部署统计模块后，看板能展示过去 7 天的订单量趋势图和步骤 1→6 各步骤的完成数漏斗。

**Acceptance Scenarios**:

1. **Given** 运营人员打开数据看板，**When** 切换时间范围为「本周」，**Then** 订单量、完成率、失败率数字刷新为本周数据。
2. **Given** 看板展示步骤转化漏斗，**When** 查看，**Then** 各步骤完成人数依次可见（步骤 1→2→3→4→5→6），直观反映漏出率。
3. **Given** 生成失败率存在异常高峰，**When** 运营人员查看「失败原因分布」，**Then** 可看到按原因分类的失败数量（如模型超时、API 限流等）。

---

### Edge Cases

- 运营人员角色无法访问「系统参数配置」时，菜单隐藏或跳转到 403 页面，不暴露敏感配置。
- 触发「重新生成」后，若该订单已有进行中的任务，系统拒绝重复提交并提示「任务已在队列中」。
- 上传 .dotx 格式模板时，文件大小超过限制（建议 20MB）时，给出明确提示，不静默失败。
- 批量导出订单列表时，若数据量超过单次限制，系统提示需缩小筛选范围或分批导出。
- 查重 API 密钥配置错误时，系统在配置保存前进行连通性测试并给出明确错误信息，不在用户使用时才暴露错误。

## Requirements *(mandatory)*

### Functional Requirements

**订单与任务**

- **FR-001**: 系统 MUST 提供订单列表，支持按时间范围、订单状态（生成中 / 写作完成 / 失败）、用户关键词、论文标题关键词筛选，并支持导出为 CSV。
- **FR-002**: 系统 MUST 在订单详情中展示：订单基本信息、支付信息、论文生成任务进度（按章）、任务日志（逐章报错信息）。
- **FR-003**: 运营人员 MUST 能够对失败订单执行「重新触发生成」，将任务重新加入后台队列。
- **FR-004**: 运营人员 MUST 能够在订单详情写入备注，备注持久化保存。
- **FR-005**: 系统 MUST 展示后台生成任务队列状态（排队中 / 执行中 / 完成 / 失败），并支持对单个任务执行取消或重试。

**格式模板与学校**

- **FR-006**: 系统 MUST 提供学校/格式模板列表，支持按学校名称、学历类型、启用状态筛选与搜索。
- **FR-007**: 运营人员 MUST 能够新建、编辑格式模板，包含：学校名称、学历类型、.dotx 文件上传、参考文献著录规范、标题层级样式、页眉页脚配置；模板支持启用/停用。
- **FR-008**: 系统 MUST 提供「用户未找到学校」提交记录列表，运营人员可查看用户上传的规范文档，并将记录标记为「待补充 / 已补充 / 已忽略」。
- **FR-009**: 运营人员将模板「启用」后，前台步骤 1 的学校搜索结果 MUST 立即包含该学校。

**用户管理**

- **FR-010**: 系统 MUST 提供用户列表，支持按注册时间、最近活跃时间、用户 ID 筛选，基础信息脱敏展示。
- **FR-011**: 系统 MUST 在用户详情中展示：基本信息（脱敏）、订单总数、完成数、失败数、最近订单列表。
- **FR-012**: 运营人员 MUST 能够对用户执行禁用操作，被禁用用户的前台登录请求被拒绝；MUST 能够配置单用户单日生成上限（限流）。
- **FR-013**: 所有用户禁用与限流操作 MUST 自动记录到操作日志。

**权限与账号**

- **FR-014**: 系统 MUST 支持多角色（超级管理员 / 运营 / 客服 / 只读），不同角色绑定不同菜单与操作权限。
- **FR-015**: 超级管理员 MUST 能够创建、编辑、禁用后台账号，并分配角色。
- **FR-016**: 系统 MUST 记录关键操作日志，包含：操作者账号、时间、操作类型、操作对象；日志支持按时间范围和操作类型筛选查询。

**配置管理**

- **FR-017**: 运营人员 MUST 能够配置查重 API（服务商地址、密钥、限流参数）并在保存前进行连通性测试。
- **FR-018**: 运营人员 MUST 能够配置引用核对接口地址与开关。
- **FR-019**: 系统 MUST 提供全局参数配置（维护模式开关、默认字数建议、文献数量建议等），修改后前台立即生效。

**数据统计**

- **FR-020**: 系统 MUST 提供数据看板，展示：订单量趋势、步骤转化漏斗（1→6）、生成成功率、失败原因分布；支持按日/周/月切换。
- **FR-021**: 系统 MUST 展示收入汇总及按版本/套餐的收入分布（仅在已商业化时展示）。

**商品与支付**

- **FR-022**: 运营人员 MUST 能够管理商品版本（基础版 / AI 率保障版 / 无限改稿版）：配置价格、描述、赠品内容；支持上下架。
- **FR-023**: 系统 MUST 提供支付记录查询，运营人员可查看每笔支付的状态与关联订单。

### Key Entities

- **Order（订单）**: 与用户微信账号关联；包含论文标题、状态、创建/完成时间、支付信息、套餐版本；关联一个 GenerationJob。
- **GenerationJob（生成任务）**: 属于订单；包含按章生成进度、各章状态与日志；可排队、执行、完成、失败；支持重试。
- **SchoolTemplate（学校格式模板）**: 学校名称、学历类型、.dotx 文件、参考文献规范、标题层级、页眉页脚配置、启用状态。
- **TemplateRequest（用户提交的学校请求）**: 用户提交的学校名称、上传的规范文档、处理状态；可关联到已建模板。
- **AdminUser（后台账号）**: 登录凭证、角色；绑定菜单/操作权限。
- **OperationLog（操作日志）**: 操作者、时间、操作类型、操作对象 ID 与内容摘要。
- **Product（商品版本）**: 版本名称、价格、赠品描述、上下架状态。
- **ApiConfig（接口配置）**: 配置类型（查重 / 引用核对）、服务地址、密钥（加密存储）、开关。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 运营人员从登录后台到找到并重试一个失败订单的全流程，可在 3 分钟内完成。
- **SC-002**: 运营人员从接收到用户学校模板需求（客服转交）到上线可搜索的新模板，可在 10 分钟内完成全部后台操作。
- **SC-003**: 所有关键操作（订单状态变更、模板上下架、用户禁用、配置修改）100% 被记录到操作日志，无遗漏。
- **SC-004**: 后台账号权限隔离有效——不同角色账号登录后，菜单与操作权限与配置完全一致，不出现越权访问。
- **SC-005**: 数据统计看板在切换时间范围时，数据刷新时间不超过 5 秒（99 百分位）。
- **SC-006**: 查重 API 配置切换后，前台步骤 6 查重请求在切换完成后立即使用新配置，无需重启服务。

### Assumptions

- 后台管理网站仅供内部运营人员使用，不对外公开；通过独立域名或路径与前台用户端隔离。
- 后台账号采用用户名 + 密码登录（非微信扫码），初始账号由系统管理员在部署时创建。
- 数据统计基于已有订单与用户数据计算，不接入独立数据仓库（OLAP），统计查询直接走主库，数据量在初期可接受。
- 商品与支付模块的「退款申请与处理」超出本期 spec 范围（产品约束：不支持退款），本期仅覆盖支付记录查询。
- 内容抽检、敏感词策略（原清单 P3 模块）不在本期 spec 范围内。
